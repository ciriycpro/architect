# DEC-026 — Rate-limit: per-tenant + Redis на горизонте мульти-тенанта

## Status

Accepted (30.05.2026). **Решение принято, реализация отложена** до этапа мульти-тенанта (v3). Текущий per-IP in-memory rate-limit (DEC-017 Уровень 0) остаётся в силе для v1 (один узел, один-два ИП через оркестратор).

## Context

В `compliance-logic` есть `RateLimitFilter` (bucket4j, token-bucket): ключ — IP клиента (`getRemoteAddr`), бакеты в памяти процесса (`ConcurrentHashMap`), группы `documents-upload 30/мин`, `clients-create 60/мин`, `metrics 120/мин`, `default 600/мин`. Это DEC-017 Уровень 0 — защита от DoS/флуда на одном узле.

Для целевой высоконагруженной SaaS-архитектуры (банки/консалтинг/аудит, мульти-тенант) текущая модель не масштабируется по двум причинам:

1. **Ключ по IP неверен для server-to-server.** Все вызовы идут с пары бэкенд-IP (оркестратор/шлюз), а не от множества конечных пользователей. Один IP-бакет душит всю платформу разом — лимит одного клиента бьёт по всем.
2. **Бакеты в памяти процесса не переживают горизонтальное масштабирование.** При нескольких подах (DEC-016 K8s) у каждого свой счётчик — фактический лимит = N×настроенный, и течёт непредсказуемо.

Запас под нагрузку даётся не увеличением числа (600 → 5000), а **сменой измерения ключа и хранилища**.

## Decision

На этапе мульти-тенанта (v3) переписать rate-limit:

1. **Ключ — per-tenant (API-key / client_id), не IP.** Каждый тенант — свой бакет, изоляция «шумного соседа».
2. **Глобальный потолок на узел** поверх per-tenant — защита ноды от суммарного флуда.
3. **Тарифные группы per-tenant**: раздельные лимиты для reads / writes / тяжёлых LLM-вызовов (summary, vision, генерация ответа банку). Лимиты — параметр тарифа клиента, не хардкод.
4. **Бакеты в Redis** (bucket4j-redis / Lettuce) — распределённое состояние, корректно при N подах.

Текущие per-endpoint группы (`documents-upload`, `clients-create` и т.д.) сохраняются как вторичное измерение поверх per-tenant.

## Триггеры реализации

Любой из:
- (а) второй платящий тенант помимо текущих ИП;
- (б) переход на горизонтальное масштабирование (2+ пода `compliance-logic`);
- (в) появление отдельного API-gateway перед сервисами.

## Consequences

**Плюсы:**
- Изоляция тенантов — лимит одного не влияет на других.
- Корректность при горизонтальном масштабировании (общий счётчик в Redis).
- Лимиты привязаны к тарифу — монетизационный рычаг (разные планы = разные квоты).

**Минусы:**
- Зависимость от Redis (ещё один компонент в SLA). Митигация: Redis всё равно нужен для distributed state на v3.
- Латентность +1 сетевой вызов на запрос к Redis. Митигация: local-cache бакетов с периодической синхронизацией.

**Что НЕ делаем сейчас (v1):** ничего не трогаем — per-IP in-memory 600/мин дефолт с запасом покрывает один узел и два ИП. Преждевременная реализация = over-engineering.
