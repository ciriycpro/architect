# 5. Mail-service как отдельный микросервис

Date: 2026-05-02

## Status

Accepted (decision) — Implementation paused at architecture fork

## Context

После DEC-002 IMAP-канал планировался через ноду N8N. После DEC-004 он временно убран. Нужно вернуть приём писем от Таирова, но N8N IMAP-нода имеет проблемы:

- Глючит на больших вложениях (>5 MB)
- mail.ru блокирует через час работы (timeout)
- Нет контроля над парсингом MIME, attachments
- Сложно дебажить — ноды живут в БД N8N

Дополнительно выявлено в ночь со 02 на 03 мая при работе с Полигоном (id `1632604d64e14e20`, 14 нод, IMAP+Drive+Sheets): нестабильность N8N IMAP-ноды, баг регистрации webhook после правок в БД ([n8n issue #16339](https://github.com/n8n-io/n8n/issues/16339)), потеря писем при переходе UNSEEN→SEEN. Полигон оставлен как полу-заглушка (см. DEC-006).

## Decision

Извлечь IMAP-приём в **отдельный микросервис на Python/FastAPI**:

- Имя: `mail-service`
- Хост: `coo:8765` (`127.0.0.1:8765`, не на 0.0.0.0)
- Стек: Python 3.10 + FastAPI + uvicorn + стандартный `imaplib`
- Endpoint v1: `GET /mail/since/<YYYY-MM-DD>` — возвращает массив писем с полями `messageId`, `from`, `email`, `fio`, `subject`, `date` (ISO), `body_text` (≤5000 символов), `attachment_names[]`. Без скачивания вложений, без OCR, без саммари.
- Документовед в N8N (или его замена в стеке `mail-stack/`) принимает данные через HTTP webhook от mail-service.

Изначально предполагался Node.js + `imap-simple`. По итогам работы 03.05 выбран Python + `imaplib` — стабильнее, проще отладка через `curl`, не требует Node.js окружения отдельно от N8N.

## Consequences

**Плюсы:**
- Полный контроль над IMAP-логикой (timeouts, retry, логирование)
- Сервис можно перезапускать независимо от N8N
- Дебаг через `journalctl` (если в systemd) или `docker logs` (если в Docker), плюс прямая проверка `curl http://127.0.0.1:8765/mail/since/...`
- Открытый путь для замены N8N полностью на стек микросервисов (см. DEC-006)
- Корректно парсит кириллицу, MIME-encoded заголовки, ФИО+email раздельно, имена вложений (включая русские)

**Минусы:**
- Новый компонент — новая точка отказа
- Нужно деплоить и мониторить
- Контракт между mail-service и потребителями нужно поддерживать
- Парсер пропускает письма с ошибкой парсинга через `try/except + continue` без счётчика ошибок — упавшие письма не попадают в выдачу и об этом узнать нельзя без отдельного логирования. Замечание к v2.

**Текущий статус реализации (на 09.05.2026).**

v1 реализован 03.05.2026 на coo:
- Путь: `/opt/mail-stack/mail-service/` (перенесено 09.05.2026 из `~/mail-stack/mail-service/` для производственной установки в стандартное место)
- Состав: `server.py` (~6 КБ, ~250 строк) + Python venv с `fastapi`, `uvicorn`, `pydantic`, `starlette`, `anyio`, `h11`
- Endpoints: `GET /health` → `{"status":"ok","service":"mail-service-v1"}`, `GET /mail/since/<YYYY-MM-DD>`
- Конфиг: hardcoded в `server.py` (IMAP host/port/user/pass) — замечание к v2 о вынесении в env

End-to-end тест 03.05.2026 14:23 UTC: 5 писем за период 01.05–03.05 успешно распарсены — корректное MIME-декодирование кириллицы, отделение ФИО от email, ISO-даты, имена вложений (включая русские, в т.ч. от Контур.Экстерн, Google, Управления Делами, Таирова с 5 JPG).

Повторный тест 09.05.2026 23:24 UTC: 6 писем за тот же период — `GET /health` отвечает, `GET /mail/since/2026-05-01` возвращает корректный JSON. Прямой IMAP-запрос мимо сервиса подтвердил счётчик: в `INBOX` за этот период действительно 6 писем (из 6462 всего в `INBOX`), остальное — в тематических папках mail.ru.

**v1 НЕ оформлен в systemd-юнит** (для сравнения: Agent Caller оформлен — `/etc/systemd/system/agent-caller.service`, active с 27.04.2026). v1 НЕ запущен в продакшен-режиме, запускался только foreground-uvicorn для тестов.

**Причина паузы — архитектурная развилка 03.05.2026 после успешного теста.** Решение «писать сразу docker-friendly код и сразу класть в Docker» обсуждалось, но не реализовано. Mail-service остался последним bare-metal сервисом перед запланированной докеризацией всего стека `mail-stack/`. Развилка зафиксирована в [DEC-006](0006-reality-check-09052026.md), решение по форме развёртывания — отложено.

**Эволюция архитектуры:**
- DEC-001: всё в N8N
- DEC-002: + IMAP в N8N (отменено)
- DEC-005: гибрид — IMAP в Python-сервисе, OCR в N8N
- DEC-006: реальный полный стек микросервисов как направление; форма развёртывания (Docker vs systemd) — pending

**Стратегический сигнал:** возможен полный переезд на стек Python-микросервисов (mail / attachment / OCR / summary / classifier / state) при дальнейшем упоре в потолок N8N. Mail-service v1 — первый камень этого стека.
