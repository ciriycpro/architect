# 5. Mail-service как отдельный микросервис

Date: 2026-05-02

## Status

Accepted

## Context

После DEC-002 IMAP-канал планировался через ноду N8N. После DEC-004 он временно убран. Нужно вернуть приём писем от Таирова, но N8N IMAP-нода имеет проблемы:

- Глючит на больших вложениях (>5 MB)
- mail.ru блокирует через час работы (timeout)
- Нет контроля над парсингом MIME, attachments
- Сложно дебажить — ноды живут в БД N8N

## Decision

Извлечь IMAP-приём в **отдельный микросервис на Node.js**:

- Имя: `mail-service`
- Хост: `coo:8765`
- Стек: Node.js + `imap-simple` библиотека
- Функции: подключение к IMAP, парсинг писем, извлечение вложений, отдача по HTTP API
- Документовед в N8N принимает данные через HTTP webhook от mail-service

## Consequences

**Плюсы:**
- Полный контроль над IMAP-логикой (timeouts, retry, логирование)
- Сервис можно перезапускать независимо от N8N
- Дебаг через `docker logs` и стандартные Node.js инструменты
- Открытый путь для замены N8N полностью на микросервисы (если зайдёт)
- mail-service v1 успешно протестирован 02.05.2026 (5 писем с 01.05, парсинг ОК)

**Минусы:**
- Новый компонент — новая точка отказа
- Нужно деплоить и мониторить
- Контракт между mail-service и N8N — нужно поддерживать

**Эволюция архитектуры:**
- DEC-001: всё в N8N
- DEC-002: + IMAP в N8N (отменено)
- DEC-005: гибрид — IMAP в Node.js, OCR в N8N

**Стратегический сигнал:** возможно полный переезд на Node.js в будущем (если N8N будет упираться в потолок).
