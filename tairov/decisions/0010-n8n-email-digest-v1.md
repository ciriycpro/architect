# 10. N8N workflow «Email Digest v1» — первая работающая автоматизация

Date: 2026-05-12

## Status

Accepted (planned for implementation)

## Context

После запуска mail-stack (mail-service v1 в продакшене 12.05.2026, attachment/parser/summary-service запланированы) встаёт вопрос: **как связать сервисы вместе в работающий end-to-end сценарий**.

Из [DEC-005](0005-microservices.md): «N8N сохраняется как оркестратор, тяжёлая логика выносится в микросервисы». Из [DEC-006](0006-reality-check-09052026.md): принципы работы с N8N — один кирпич за раз, только UI (не SQL-патчи), простые Code-ноды, всё сложнее IF в Python-сервис.

Цель v1 — **минимальная работающая автоматизация для Таирова**, которая:
1. Закрывает обещание «сборщик писем со сводкой» (от 03.05.2026)
2. Заменяет Полигон (полу-заглушка, теряет письма)
3. Демонстрирует работу всего стека: mail → attachment → parser → summary → доставка
4. Служит первым кейсом для портфолио и собеседований

## Decision

Создаётся N8N workflow `Email Digest v1` со следующей структурой:

### Триггеры — два параллельных

```
[Schedule: 09:00 MSK каждый день] ──┐
                                    ├──► общая логика
[Webhook: POST /digest-now]    ─────┘
```

Schedule даёт автоматический режим. Webhook — для ручного запуска (тест, ad-hoc запрос Таирова через Telegram-команду, дебаг). Оба триггера ведут в одну и ту же ветку обработки.

### Полная цепочка нод

```
[Trigger: Schedule 09:00 MSK]    [Trigger: Webhook /digest-now]
              │                              │
              └──────────┬───────────────────┘
                         ↓
         [HTTP GET mail-service/mail/since/<вчера>?limit=50]
                         ↓
         [IF: писем > 0]
              ├─ false → [Telegram: "Сегодня писем не было"] → END
              ↓ true
         [Code: for each message → collect attachments[]]
                         ↓
         [Loop over attachments]:
              [HTTP POST attachment-service/download]
              [HTTP POST parser-service/parse]
                         ↓
         [Code: собрать messages[] с parsed_attachments внутри]
                         ↓
         [HTTP POST summary-service/summary]
                         ↓
         [Parallel split]:
              ├─ [Sheets append → лист «Дайджест»]
              └─ [Telegram message → Таирову (chat_id 1257818936)]
                         ↓
         END
```

**Итого нод:** 7-8 шт (без учёта параллельной развилки).

### Соответствие принципам DEC-006

| Принцип | Соблюдение |
|---|---|
| Один кирпич за раз | ✅ Workflow собирается по нодам последовательно, после каждой — проверка вручную |
| Никаких SQL-патчей | ✅ Все правки через UI N8N |
| Sheets только append | ✅ Используется `appendOrUpdate=append`, без `update` |
| Простые Code-ноды | ✅ Две Code-ноды: «собрать attachments» и «собрать messages», обе по 5-10 строк |
| Всё сложнее IF — в Python | ✅ Вся логика парсинга, классификации, саммари — в микросервисах. В N8N только трансляция данных |
| Экспорт перед правкой | ✅ После каждой добавленной ноды — экспорт JSON в `tairov/workflows/email-digest-v1-<timestamp>.json` |
| IMAP-нода N8N нестабильна | ✅ IMAP вынесен в mail-service, в N8N используется HTTP GET |

### Доставка результата — два канала

**Канал 1: Google Sheets.** Лист «Дайджест» в книге `13SMWzIiwDVRc1eYKJcGm1a-R__7Hbc1hvChTXvZhfsg`. Колонки: дата, количество писем, summary_markdown, токены, стоимость, статус.

**Канал 2: Telegram.** Сообщение Таирову (chat_id `1257818936`). Используется `summary_telegram` (plain-text без markdown — Telegram не везде корректно рендерит).

Дублирование намеренное:
- Sheets — архив, поиск, ретроспектива
- Telegram — оперативная доставка, утренний пуш

### Деактивация Полигона

Одновременно с активацией `Email Digest v1` в production:
- `Полигон` (workflow id `1632604d64e14e20`) → `active=0` через UI N8N
- Не удалять. Наблюдение 30 дней
- После 30 дней без инцидентов в Email Digest → экспорт JSON Полигона в `tairov/workflows/archive/` и удаление из N8N БД

Это митигирует риск из DEC-006: Полигон полу-заглушка, теряет письма. Email Digest опирается на mail-service v1, который читает почту deterministично и без потерь.

## Consequences

**Плюсы:**
- Первая работающая end-to-end автоматизация для Таирова
- Все сервисы стека (mail/attachment/parser/summary) проходят боевую проверку через реальный поток
- Schedule + Webhook покрывает оба режима использования (автомат + ручной)
- Доставка в два канала — резервирование (если Telegram упал, Sheets всё равно сохранит)
- Соответствует всем 7 принципам DEC-006

**Минусы:**
- 7-8 нод в N8N — относительно сложный workflow. Митигация: каждая нода делает одно простое действие, нет вложенных Code-нод
- Зависимость от 4 микросервисов сразу — если хоть один упал, дайджест не уйдёт. Митигация: каждый сервис имеет `/health` endpoint, мониторинг отдельно (планируется в v2 — DEC будущий)
- Стоимость одного дайджеста: ~$0.008 (Haiku 4.5) + ~$0.002 (Qwen3-VL для вложений если есть) = ~$0.01/день = ~30 руб/мес. Окупается чеком Таирова с большим запасом

**Открытые вопросы для v2:**
- Когда добавить классификацию писем по приоритету (требует действий / информационное / спам) — это отдельный `classifier-service`, v2 направление
- Когда добавить детекцию дедлайнов из писем (например, «оплатить до 15.05») — `deadline-extractor-service`, тоже v2
- Когда добавить персонализацию дайджеста по тону / ключевым словам Таирова — после статистики предпочтений
- Когда добавить мониторинг health всех сервисов и алертинг в Telegram при их падении — `monitoring-service`, v2
- **Mail Check On-Demand workflow** — Telegram-кнопка «Проверить почту» в @CallerBaby666Bot → ad-hoc проверка с момента последнего обращения (а не за фиксированный день). Требует:
  - `state-service` (или SQLite/JSON-файл) для tracking `last_check_timestamp` per-user
  - Второй N8N workflow с триггером Telegram-callback (или Webhook /check-now)
  - Другой промпт для summary-service: короткий ответ в Telegram («с последней проверки X часов назад пришло N писем, основные:...») вместо длинного дайджеста
  - При отсутствии новых писем — сообщение «новых писем нет, последняя проверка YYYY-MM-DD HH:MM»
  - Mail-stack (mail/attachment/parser/summary) переиспользуется без изменений — отличается только оркестратор
  Заслуживает отдельного ADR на v2 (предполагаемый DEC-013 «Mail Check On-Demand workflow»). В DEC-010 v1 — только Schedule-режим за полный день.

**Стратегический сигнал:** Email Digest v1 — это **прототип паттерна «сервисный конвейер с N8N-оркестрацией»**, который масштабируется горизонтально на любые задачи документооборота МСП (договоры, акты, ИФНС-требования). После Таирова — паттерн переиспользуется на следующих клиентах. **On-demand интерфейс через Telegram-кнопку** — следующий шаг превращения mail-stack в полноценный AI-ассистент: пассивный дайджест по расписанию + активный диалог по запросу.
