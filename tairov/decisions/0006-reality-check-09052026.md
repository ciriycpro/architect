# 6. Сверка архитектуры с реальностью на coo (09.05.2026)

Date: 2026-05-09

## Status

Accepted

## Context

При запуске нового чата 09.05.2026 для актуализации репозитория `ciriycpro/architect` обнаружены систематические расхождения между декларативными артефактами (DSL + ADR) и фактическим состоянием сервера coo. Артефакты разрабатывались в период 27.04.2026 – 02.05.2026 как декларативные решения; реализация частично отстала, частично пошла другим путём, частично остановлена на архитектурной развилке.

Этот ADR — снимок реальности на 09.05.2026 и фиксация принципов работы с N8N, выработанных в ходе инцидента 02–03.05.2026.

## Decision

### По существующим ADR
- **DEC-003** — Status остаётся `Cancelled`. Дополнен раздел Consequences с описанием артефакта в БД (39 нод, id `2b0af1a8c92c424f`). Артефакт сохранён до выхода mail-service в продакшен.
- **DEC-004** — Status изменён на `Accepted (decision) — Not implemented as of 09.05.2026`. В Consequences добавлен факт: в N8N нет ни одного workflow, реализующего описанный 12-нодовый пайплайн. Полигон выполняет другую функцию.
- **DEC-005** — Status изменён на `Accepted (decision) — Implementation paused at architecture fork`. В Consequences зафиксировано: v1 реализован, протестирован, перенесён в `/opt/mail-stack/mail-service/`, но не оформлен в systemd и не докеризован. Причина — развилка по форме развёртывания.

### По DSL и документации
- **`tairov/workspace.dsl`** — переписывается под фактическое состояние. Добавляется контейнер `polygon` (active workflow в N8N). Контейнеры `documentoved` и `controller` сохраняются в DSL с пометкой в description «not implemented (planned)», чтобы DSL читался как карта замысла, но честно отражал реальность. См. сам DSL для деталей.
- **`tairov/docs/01-overview.md`** — расширен от 4-строчной заглушки до Service Blueprint реальности на 09.05.2026 с явным разделением «Что работает» / «Что в работе» / «Что не реализовано».

### Открытая развилка: форма развёртывания `mail-stack/`

На 03.05.2026 после успешного теста v1 рассматривались три варианта:

1. **Чисто systemd по образцу Agent Caller.** Юнит `mail-service.service` в `/etc/systemd/system/`, запуск bare metal, доступ из N8N (в Docker) через `host.docker.internal:8765` или IP `docker0` bridge.
2. **Сразу Docker через `docker-compose`.** Mail-service в собственном контейнере в сети `n8n_default`, доступ из N8N по hostname `http://mail-service:8765`. Подразумевает написание Dockerfile (~8 строк), `requirements.txt`, выноса конфига в env, listen на `0.0.0.0:8765` внутри контейнера.
3. **Гибрид:** mail-service в systemd сейчас (быстрый старт), все следующие сервисы (`attachment-service`, `ocr-service`, `summary-service`, `classifier-service`, `state-service`) — сразу в Docker. Перенос mail-service в Docker — позже, когда стек окрепнет.

**Решение по выбору варианта — отложено.** Эта работа сегодня — синхронизация документации, не проектирование. Развилка фиксируется как открытый вопрос. К решению вернуться в отдельной сессии после расстановки приоритетов.

## Consequences

**Плюсы:**
- DSL и ADR начинают описывать реальность, а не желаемое. Любая новая сессия Claude или инженера получит честную картину за минуту чтения.
- Решения остаются `Accepted` (DEC-004, DEC-005 не отменены), но честно маркированы статусом реализации.
- Зафиксированы рабочие принципы N8N, выработанные кровью.
- Зафиксирован факт существования mail-service v1 как working code — иначе через месяц можно было снова забыть, где он лежит.

**Минусы:**
- DSL временно «беднеет» в смысле визуализации — реальный рабочий контур состоит из Полигона (полу-заглушка) и Agent Caller. Mail-service есть, но не запущен. Документовед v2 и Контроллер только в плане. Это болезненно смотреть, но честно.
- Архитектурная развилка по форме развёртывания не решена в этом ADR. Без решения нельзя двинуться дальше по `mail-stack/`. Решение требует отдельной сессии.
- Полу-работающий Полигон в активе — риск молчаливой потери данных (часть писем пишется в Sheets, часть нет). Для production этого недостаточно.

## Working principles for N8N (выстрадано в инциденте 02–03.05.2026)

Эти принципы выработаны в ходе четырёхчасового марафона по починке Полигона. Должны соблюдаться при любой работе с N8N в этом проекте:

1. **Один кирпич за раз.** Не собирать workflow из 14+ нод сразу. Добавлять по одной ноде, сохранять через UI, тестировать 5 ручных запусков, переходить к следующей. Полигон-Франкенштейн (39 нод, DEC-003) и Полигон v2 (14 нод, неработоспособен) — следствия нарушения этого правила.

2. **Никаких SQL-патчей в БД N8N.** Только UI. Любое `UPDATE workflow_entity SET nodes=?` через прямую запись в `database.sqlite` ломает регистрацию webhook ([n8n issue #16339](https://github.com/n8n-io/n8n/issues/16339)) и портит экранирование expressions (markdown-рендер в чатах ломает `$json.email` → `$[json.email](http://json.email)`, и нода молча отдаёт пустую строку, что вызывает «Could not get parameter» далее по цепочке).

3. **Sheets-ноды на старте — только `append`.** Никаких `appendOrUpdate` с `matchingColumns` пока поток данных не стабилен. Дедупликацию на старте делать на уровне источника или отдельной Code-нодой по `messageId`.

4. **Code-ноды простые.** Каждая делает ОДНУ трансформацию. Никаких 200-строчных pipeline-ов в одной Code — выносить в Python-микросервис в `mail-stack/`.

5. **Всё, что в N8N сложнее одной IF-ноды — выносить в Python-сервис.** N8N — оркестратор, не процессор. Парсинг MIME, OCR, классификация, дедупликация, поиск отправителя — это работа Python-сервисов. N8N делает только: триггер → HTTP вызов → Sheets append → HTTP вызов → Telegram message.

6. **Перед каждым новым кирпичом — экспорт текущего workflow в JSON.** Чтобы откатиться, если новый кирпич сломал.

7. **IMAP-нода N8N — нестабильна.** Использовать только в качестве временной заглушки. Production-канал — собственный `mail-service`.

## Open questions

- **Форма развёртывания `mail-stack/`** (см. раздел «Открытая развилка» выше). Требует отдельного решения.
- **Когда удалять inactive `Документовед` (39 нод).** Привязано к выходу mail-service в прод.
- **Нужен ли Контроллер как отдельный N8N workflow или как Python-сервис.** Логика «Schedule + сверка по ИНН → проблемы no_contract / draft_only / expired → HTTP→Caller». В DSL он есть, в реальности — нет.
- **Что делать с Полигоном.** Вариант (а) — чинить и довести до рабочего состояния. Вариант (б) — оставить как есть, ждать замены mail-service'ом. Вариант (в) — деактивировать (active=0) до завершения mail-stack. Текущий выбор владельца — (б).
